# Typed declarations

A struct tag is a string. Nothing checks that `quer:"page"` is a typo, that
`maxx=100` is not a rule, or that a length rule was written on an integer —
and no editor can suggest what belongs inside the quotes.

`DefineInput` says the same thing in code:

```go
// Plain struct. Fields and types, nothing else.
type ListItemsInput struct {
	Page   int
	Limit  int
	Search string
	Sort   []string
	Token  string
}

var listItemsInput = tork.DefineInput(func(b *tork.InputBuilder, in *ListItemsInput) {
	b.Query.Int(&in.Page, "page").Default(1).Min(1)
	b.Query.Int(&in.Limit, "limit").Default(20).Range(1, 100)
	b.Query.String(&in.Search, "search").MaxLen(200)
	b.Query.Strings(&in.Sort, "sort").MaxItems(3).OneOf("name", "-created")
	b.Header.String(&in.Token, "X-Token").Required()
})
```

**The handler does not change.** It takes the struct, exactly as before:

```go
func listItems(ctx context.Context, in ListItemsInput) ([]ItemResponse, error)
```

The declaration is found by the struct's type, the same way `orm.DefineTable`
registers a table by its row type. Assigning it to a package-level variable is
what runs it; nothing needs to be done with the value.

## What the compiler now catches

```go
b.Query.String(&in.Page, "page")
// cannot use &in.Page (value of type *int) as *string value

b.Query.Int(&in.Page, "page").MaxLen(10)
// p.MaxLen undefined (type *tork.IntParam has no field or method MaxLen)

b.Query.Int(&in.Limit, "limit").Default("twenty")
// cannot use "twenty" (untyped string constant) as int value

b.Query.Time(&in.Since, "since").After("yesterday")
// cannot use "yesterday" (untyped string constant) as time.Time value
```

And typing `b.` offers the five sources, `b.Query.` offers the field types, and
`.Min(1).` offers only the rules an integer has.

## How a field is named

Every builder method takes a pointer to the field it fills. The pointer comes
from the template struct the callback is given, and which field it names is
worked out once, when the declaration runs, by matching both the address and
the type.

That template exists only to have its fields' addresses taken. It is never the
value a handler receives, and nothing written into it means anything.

## The sources

| Source | Reads |
|---|---|
| `b.Path` | the route's wildcards |
| `b.Query` | the URL's query string |
| `b.Header` | request headers |
| `b.Cookie` | request cookies |
| `b.Form` | a urlencoded or multipart body |
| `b.JSONBody(&in.Body)` | the whole request body |

Each source offers the same field types:

```go
b.Query.String(&in.Name, "name")
b.Query.Int(&in.Count, "count")
b.Query.Int64(&in.Total, "total")
b.Query.Float64(&in.Ratio, "ratio")
b.Query.Bool(&in.Enabled, "enabled")
b.Query.Time(&in.At, "at")             // RFC 3339
b.Query.Duration(&in.Wait, "wait")     // 1h30m
b.Query.Strings(&in.Tags, "tags")
b.Query.Ints(&in.Sizes, "sizes")
```

Each also has an `Optional` form, for a field that has to remember whether it
was sent at all — `OptionalString`, `OptionalInt`, `OptionalInt64`,
`OptionalFloat64`, `OptionalBool`, `OptionalTime`. Their rules are checked
against the value when there is one, and skipped when there is not.

A type not in that list — a pointer, an `encoding.TextUnmarshaler`, an uploaded
file — is declared with a [struct tag](/docs/2.fundamentals/4.query-parameters)
instead. The two forms exist side by side, and a struct picks one.

## Every field has to be declared

A field nobody mentioned would stay zero forever, which is the one way this
form could be quieter than tags. So it is checked:

```
items/input.go:14: field Search of items.ListItemsInput has no binding declared
```

Declaring the same field twice is refused for the same reason.

## Bodies

Rules for a body's fields are declared separately, because they belong to the
body type wherever it is used:

```go
type CreateItemBody struct {
	tork.JSONBody
	Name  string `json:"name"`
	Price int64  `json:"price"`
}

var createItemBody = tork.DefineBody(func(b *tork.BodyRules, in *CreateItemBody) {
	b.String(&in.Name).Required().Trim().Range(2, 120)
	b.Int64(&in.Price).NonNegative()
})
```

`BodyRules` offers the same field types as a `Source`, minus the name — and
descends: a nested struct with its own rules is checked through, and two fields
can be compared with a whole-document `Check`. See
[Validation](/docs/2.fundamentals/10.validation).

Only the rules move into code. The wire name stays in the `json` tag, because
that is what `encoding/json` reads to decode the document in the first place,
and saying it twice would be one more thing that can disagree.

The rules apply wherever the type is used — taken directly by a handler, or as
a field of an input struct:

```go
type UpdateItemInput struct {
	ItemID string         `path:"item_id"`
	Body   CreateItemBody `body:"json"`
}
```

Which is also why a body declared with `b.JSONBody(&in.Body)` needs no rules of
its own — they came with the type.

## Choosing between the two forms

Tags are terser and stay the right answer for a struct that only names things:

```go
type ReadItemInput struct {
	ItemID string `path:"item_id"`
}
```

Reach for `DefineInput` when a struct has rules, defaults worth checking, or
enough fields that a typo would be easy to miss. A struct is declared one way
or the other — doing both is refused, since there would be two answers to what
it means.

# Responses

A handler's success value is still exactly what [Handlers](/docs/2.fundamentals/2.handlers)
already says: `T` in `(T, error)` is serialized as JSON with status 200, and an
error-only handler answers 204 when it succeeds. Nothing about that changes,
and nothing below is required reading to write an ordinary handler.

The five types on this page exist for the rest: a different status, extra
headers, a file download, a body written as it is produced, or a redirect. You
reach for one only when a plain `T` is not the answer.

## Response — a status and headers of your own

`Response[T]` is a JSON body like any other, with a status and headers a
handler chooses instead of the framework assuming 200:

```go
func createItem(ctx context.Context, in CreateItemInput) (tork.Response[Item], error) {
	item := save(in)
	return tork.Respond(http.StatusCreated, item).WithLocation("/items/" + item.ID), nil
}
```

`Respond` and the struct literal reach the same result — `Respond` exists
because a generic function infers `T` from its argument and a generic struct
literal does not:

```go
tork.Response[Item]{Status: http.StatusCreated, Body: item}
```

`Status` left at zero resolves to 200, the same answer a plain `T` gives, so a
handler that only wants to add a header does not have to name a status too.
`WithHeader` adds any header; `WithLocation` is `WithHeader("Location", …)`
spelled out, for the common case of naming the resource just created:

```go
tork.Respond(http.StatusCreated, item).
	WithLocation("/items/" + item.ID).
	WithHeader("X-Request-ID", requestID)
```

## RawResponse — bytes, your content type

For a body that is not JSON at all — CSV, a legacy XML feed, bytes read from
somewhere else:

```go
func exportCSV(ctx context.Context, in ExportInput) (tork.RawResponse, error) {
	return tork.Raw("text/csv", buildCSV(rows)), nil
}
```

`WithStatus` and `WithHeader` chain onto it the same way `Response`'s do.

## FileResponse — a download

For a body read from an `io.Reader` rather than held in memory at once, with a
filename that becomes a download's suggested name rather than a JSON field:

```go
func downloadInvoice(ctx context.Context, in InvoiceInput) (tork.FileResponse, error) {
	f, size := openInvoice(in.InvoiceID)
	return tork.File(in.InvoiceID+".pdf", f).WithSize(size), nil
}
```

The content type is guessed from the filename's extension when `WithContentType`
does not set one, falling back to `application/octet-stream` when nothing
recognises it. `WithSize` sets `Content-Length` for a reader whose length is
known ahead of time; left unset, the response is written without one.

`Content-Disposition` goes through the standard library's `mime.FormatMediaType`
rather than a hand-rolled string, so a filename that needs quoting gets it
correctly.

## Stream — a body written as it goes

For a body too large to hold in memory, or one whose next chunk is not ready
until the last one has already gone out:

```go
func tailLog(ctx context.Context, in TailInput) (tork.Stream, error) {
	return tork.Streamed("text/plain", func(w io.Writer) error {
		return followLog(ctx, in.Path, w)
	}), nil
}
```

The writer passed to the callback flushes after every write the underlying
connection can flush at all, so the body reaches the client as it is
produced rather than buffered behind the handler's return. A connection that
cannot flush early still gets a complete body once the callback returns — that
is a fallback, not a failure.

## Redirect — a Location and no body

301, 302, 303, 307, and 308 disagree about whether a move is permanent and
whether a client may change its method following it, so a bare status number
would say neither:

```go
func legacyItem(ctx context.Context, in LegacyItemInput) (tork.Redirect, error) {
	return tork.PermanentRedirect("/items/" + in.ItemID), nil
}
```

| Constructor | Status | Meaning |
|---|---|---|
| `MovedPermanently(url)` | 301 | permanent, method may change |
| `Found(url)` | 302 | temporary, method may change |
| `SeeOther(url)` | 303 | always fetch `url` with GET |
| `TemporaryRedirect(url)` | 307 | temporary, method preserved |
| `PermanentRedirect(url)` | 308 | permanent, method preserved |
| `RedirectTo(status, url)` | yours | for a status already in hand |

`SeeOther` is the one a `POST` handler usually wants when it sends the client
on to a result rather than answering with one directly.

## A response that fails partway

`Response` and `RawResponse` always know their whole body before writing
anything, the same discipline a plain `T` follows — a value that cannot be
encoded, or a status that means nothing (an out-of-range `Redirect`), is still
a clean failure, answered with the same error envelope any other clean failure
gets.

`FileResponse` and `Stream` cannot make that promise: a reader or a callback
can fail after headers and part of the body are already on the wire, and there
is no way to take either back. When that happens the response is only ever
what had already been sent — the failure itself is logged, not served, because
answering again would either be rejected outright or corrupt what a client
already received.

## Writing your own

Every type above implements `Responder`:

```go
type Responder interface {
	Spec() ResponseSpec
	WriteResponse(w http.ResponseWriter, r *http.Request) error
}
```

`Spec` is asked once, at build, against a zero value of the type — which is
why it can only ever report what is true of every instance, such as a fixed
status, and not what a particular request will actually send. `WriteResponse`
is asked once per request, against the real value, and does the writing. A
result type that implements this interface is found and used the same way the
five built in here are; nothing else has to know about it.

## Documenting more than the type says

A `Response[T]`'s real status is chosen per request, and `Redirect`'s is one
of five — neither can say more than a default from a zero value. `Responds`
fills in the rest for the OpenAPI document a later phase builds:

```go
router.GET("/{item_id}", readItem,
	tork.Responds[Item](200, "the item"),
	tork.Responds[tork.Error](404, "no such item"),
)
```

`Throws` does the same for an error a handler or a dependency returns rather
than one the framework already recognises:

```go
router.GET("/{item_id}", readItem, tork.Throws[OutOfStockError]())
```

Both are metadata only — declaring them changes nothing about what a request
actually gets back. Declared on a router, they reach every route underneath;
a route-level `Responds` for a status the router already declared replaces
the router's description for it, the same way every other inherited option
works. `Throws` has no status to collide over, so declaring the same type
twice is not a mistake — an operation legitimately fails in more than one way.

# Errors

Every failing request answers with the same shape:

```json
{
  "code": "RESOURCE_NOT_FOUND",
  "message": "No order found with ID 'ord_98765'.",
  "status": 404,
  "timestamp": "2026-07-23T16:37:23Z",
  "path": "/api/v1/orders/ord_98765",
  "details": {"resource": "Order", "identifier": "ord_98765"}
}
```

One shape for everything is the point. A client that can read one error can read
all of them: `code` says what went wrong in a form code can switch on, `message`
says it in a form a person can read, and `details` carries whatever that
particular failure needs. Only `details` changes shape.

`timestamp` and `path` are filled in when the response is written, not when the
error is constructed — a handler has no business reading the clock, and does not
know the path it was reached by.

## Returning one

`tork.Error` is an ordinary Go error, so a handler returns it:

```go
func readItem(ctx context.Context, in ReadItemInput) (ItemResponse, error) {
	item, err := store.Find(ctx, in.ItemID)
	if err != nil {
		return ItemResponse{}, err
	}
	if item == nil {
		return ItemResponse{}, tork.NotFound("Item", in.ItemID)
	}
	return toResponse(item), nil
}
```

| Constructor | Status | Code |
|---|---|---|
| `BadRequest(message)` | 400 | `BAD_REQUEST` |
| `Unauthorized(message)` | 401 | `UNAUTHORIZED` |
| `Forbidden(message)` | 403 | `FORBIDDEN` |
| `NotFound(resource, id)` | 404 | `RESOURCE_NOT_FOUND` |
| `Conflict(message)` | 409 | `CONFLICT` |
| `ValidationFailed(fields...)` | 400 | `VALIDATION_ERROR` |
| `RateLimited(message, details)` | 429 | `RATE_LIMIT_EXCEEDED` |
| `Internal()` | 500 | `INTERNAL_ERROR` |
| `NewError(status, code, message)` | yours | yours |

`WithDetails` and `WithMessage` return the error, so a constructor and its
details read as one expression:

```go
return tork.Conflict("That item is out of stock.").
	WithDetails(map[string]string{"sku": sku})
```

## Details

`details` is whatever the failure needs, and is left out when it needs nothing.
The convention: a list of field problems for anything that rejected input, an
object for everything else.

```go
// Rejected input
tork.ValidationFailed(
	tork.FieldError{Field: "cardNumber", Issue: "card_number_invalid", Message: "Card number must be 16 digits long."},
	tork.FieldError{Field: "zipCode",    Issue: "field_required",      Message: "Zip code is required for billing."},
)

// Something structured
tork.RateLimited(
	"You have exceeded the rate limit of 100 requests per minute.",
	tork.RateLimitDetails{Limit: 100, Window: "1m", RetryAfterSeconds: 42, ResetAt: tork.Timestamp(resetAt)},
)
```

```json
{
  "code": "RATE_LIMIT_EXCEEDED",
  "message": "You have exceeded the rate limit of 100 requests per minute.",
  "status": 429,
  "timestamp": "2026-07-23T16:37:23Z",
  "path": "/api/v1/search",
  "details": {"limit": 100, "window": "1m", "retryAfterSeconds": 42, "resetAt": "2026-07-23T16:38:05Z"}
}
```

## Field errors

A `FieldError` has three keys. `Issue` is the half client code acts on, so it is
a constant; `Message` is prose and may be reworded at any time.

Fields are named by where they came from. A body field keeps its bare dotted
path, and a parameter carries its location, so the two can never be confused:

```
body:    billing.zipCode, cardNumber
query:   query.limit
header:  header.X-Token
cookie:  cookie.session
path:    path.item_id
```

The issues the framework itself produces: `field_required`, `invalid_integer`,
`invalid_number`, `invalid_boolean`, `invalid_datetime`, `invalid_duration`,
`invalid_format`, `invalid_type`, `invalid_json`, `body_required`,
`unknown_field`. They are exported as `tork.IssueInvalidInteger` and so on.

## Your own error types

An error that implements `HTTPError` answers for itself:

```go
type OutOfStockError struct {
	SKU string
}

func (e OutOfStockError) Error() string { return "out of stock: " + e.SKU }

func (e OutOfStockError) HTTPError() tork.Error {
	return *tork.Conflict("That item is out of stock.").
		WithDetails(map[string]string{"sku": e.SKU})
}
```

One method rather than separate status and body methods, so the two cannot
disagree: the status served is the status in the body.

It is found through `errors.As`, so wrapping does not hide it:

```go
return fmt.Errorf("loading the basket: %w", OutOfStockError{SKU: sku})
```

Only the `HTTPError` interface has to be imported from Tork, so a domain package
can define its own errors without depending on the transport layer for anything
else.

## Mapping a domain error

`OnError` translates an error into one the framework understands, in one place,
without the domain package knowing about HTTP at all:

```go
app := tork.New(
	tork.OnError(func(err error) *tork.Error {
		if errors.Is(err, orm.ErrNoRows) {
			return tork.NotFound("Resource", "")
		}
		return nil
	}),
)
```

Return nil for anything you have nothing to say about. Mappers are tried in the
order declared and the first non-nil answer wins.

They see **every** error, including ones that already implement `HTTPError`, so
an application can override anything — its own errors, and the framework's.

## Unrecognised errors

An error nothing recognises answers 500 and says nothing else:

```json
{
  "code": "INTERNAL_ERROR",
  "message": "An unexpected error occurred.",
  "status": 500,
  "timestamp": "2026-07-23T16:37:23Z",
  "path": "/api/v1/items/42"
}
```

The original is logged in full. It is not served because an unrecognised error's
message is as likely to be a connection string as an explanation.

The logger is `slog.Default()` unless you replace it:

```go
app := tork.New(tork.Logger(logger))
```

## Errors the framework raises itself

| Situation | Status | Code |
|---|---|---|
| No route matches the path | 404 | `ROUTE_NOT_FOUND` |
| The path exists, the method does not | 405 | `METHOD_NOT_ALLOWED` |
| A request could not be bound | 400 | `VALIDATION_ERROR` |
| The body is not the media type expected | 415 | `UNSUPPORTED_MEDIA_TYPE` |
| The body is over `MaxBodyBytes` | 413 | `PAYLOAD_TOO_LARGE` |
| A handler panicked, or failed unrecognisably | 500 | `INTERNAL_ERROR` |

`ROUTE_NOT_FOUND` and `RESOURCE_NOT_FOUND` are both 404 and are deliberately
different codes: one means the client asked for something this API does not
have, the other that it asked correctly for a thing that does not exist. A
client retrying the second and giving up on the first is behaving sensibly, and
can only do so if they say so.

A 405 also carries an `Allow` header, and an `OPTIONS` request is answered from
the methods a path declares without a handler of its own.

## Replacing the format entirely

For an API that has to answer in a shape it did not choose:

```go
app := tork.New(tork.WriteErrorsWith(
	func(w http.ResponseWriter, r *http.Request, e tork.Error) error {
		w.Header().Set("Content-Type", "application/problem+json")
		w.WriteHeader(e.Status)
		return json.NewEncoder(w).Encode(toRFC9457(e))
	},
))
```

The error is complete by the time it arrives — status, code, message, timestamp,
and path all filled in — so the writer only has to serialize it.

## Timestamps

`tork.Timestamp` is a `time.Time` that reads as RFC 3339 to the second, in UTC:

```
"2026-07-23T16:37:23Z"
```

Truncating at the boundary is what makes two servers agree and a golden file
possible; `time.Time` would otherwise marshal with whatever fractional precision
it happened to carry. It round-trips, so a client can decode an error body back
into the same type.

For tests, `tork.Clock` holds time still:

```go
app := tork.New(tork.Clock(func() time.Time { return fixed }))
```

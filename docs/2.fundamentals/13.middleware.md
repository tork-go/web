# Middleware

Middleware is the ecosystem's own shape:

```go
type Middleware = func(http.Handler) http.Handler
```

It is a type alias, not a defined type, so anything already written for
`net/http` composes without adaptation — and anything you write for Tork works
outside it.

```go
func RequestID(next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		id := r.Header.Get("X-Request-ID")
		if id == "" {
			id = newID()
		}
		w.Header().Set("X-Request-ID", id)
		next.ServeHTTP(w, r.WithContext(context.WithValue(r.Context(), requestIDKey{}, id)))
	})
}
```

## Attaching it

`Use` works at all three levels, and is inherited:

```go
app := tork.New(tork.Use(RequestID, Recoverer))

router := tork.NewRouter(
	tork.Prefix("/admin"),
	tork.Use(RequireAdmin),
)

router.GET("/audit", readAudit, tork.Use(RateLimit))
```

## Order

Middleware runs outermost first: the application's, then the router's, then the
route's, and within one `Use` in the order written.

```go
app := tork.New(tork.Use(first, second))
router := tork.NewRouter(tork.Use(third))
router.GET("/", handler, tork.Use(fourth))
```

```
first -> second -> third -> fourth -> handler
```

## What it can do

Middleware sits outside everything the framework does for a request, which is
both its strength and its limit. It runs before any binding, so it cannot see
the decoded input; it can answer a request without the handler ever running:

```go
func RequireAdmin(next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		if !isAdmin(r) {
			w.WriteHeader(http.StatusForbidden)
			return
		}
		next.ServeHTTP(w, r)
	})
}
```

Note what that costs: writing to the `ResponseWriter` directly bypasses the
error envelope, so a client gets a bare 403 with no body. A
[dependency](/docs/2.fundamentals/14.dependencies) is the better place for
authorization: a guard returning `tork.Forbidden(...)` answers with the
envelope like everything else, and runs after binding rather than before it.
Middleware that refuses a request should write the envelope itself if the API's
clients expect one.

## Passing values to handlers

The request's context is the way through, since a handler taking
`context.Context` gets the one middleware built:

```go
func handler(ctx context.Context, in ListItemsInput) (Response, error) {
	id, _ := ctx.Value(requestIDKey{}).(string)
}
```

A [dependency](/docs/2.fundamentals/14.dependencies) is the typed alternative:
a `Principal` decoded once and injected as a parameter, rather than an `any`
fished back out of a context. The context is still the way through for a value
a middleware — not the framework — produced.

## Panics

The framework already recovers panics from handlers, logs them, and answers 500
with the error envelope, so a recovery middleware is not required. One is still
useful if you want panics from *other middleware* handled the same way — the
framework's recovery is inside the middleware chain, not outside it.

# Validation

Rules are declared with the [typed builders](/docs/2.fundamentals/9.typed-declarations),
which is what makes them checkable: `MaxLen` exists on a string and does not
exist on an integer, so writing it on the wrong field does not compile.

The same builders declare a parameter and a body field, so every rule below
works in both:

```go
var listItemsInput = tork.DefineInput(func(b *tork.InputBuilder, in *ListItemsInput) {
	b.Query.Int(&in.Page, "page").Default(1).Min(1)
	b.Query.String(&in.Search, "search").Trim().MaxLen(200)
	b.Header.String(&in.Token, "X-Token").Required()
})

var createItemBody = tork.DefineBody(func(b *tork.BodyRules, in *CreateItemBody) {
	b.String(&in.Name).Required().Trim().Range(2, 120)
	b.String(&in.Email).Required().Trim().ToLower().Email()
	b.Int64(&in.Price).NonNegative()
})
```

## Transforms

A transform is not a check — it is what the value becomes. Transforms run
**before** every rule, so a length rule measures what was meant rather than
what was pasted, and the handler receives the cleaned value.

| Transform | Effect |
|---|---|
| `Trim()` | drops leading and trailing whitespace |
| `Collapse()` | trims, and reduces runs of inner whitespace to one space |
| `ToLower()` / `ToUpper()` | folds the case |
| `UTC()` *(times)* | moves the value to UTC |

```go
b.String(&in.Email).Trim().ToLower().Email()
b.String(&in.Title).Collapse().MaxLen(80)
b.Time(&in.At).UTC()
```

`Trim().Required()` is worth knowing as a pair: trimming a field of nothing but
spaces leaves it empty, and `Required` then says so.

## The rules

**Strings** — `Required` `Default` `MinLen` `MaxLen` `Range` `Len` `NotBlank`
`OneOf` `Pattern` `Contains` `HasPrefix` `HasSuffix`

**Formats** — `Email` `UUID` `URL` `IP` `IPv4` `IPv6` `CIDR` `Hostname`
`Semver` `Slug` `CreditCard` `Base64` `JSON` `Alpha` `Alphanumeric` `Numeric`
`ASCII` `Hex` `Lowercase` `Uppercase`

**Integers** (`Int`, `Int64`) — `Required` `Default` `Default64` `Min` `Max`
`Range` `Positive` `NonNegative` `Negative` `NonPositive` `OneOf` `MultipleOf`

**Floats** — `Required` `Default` `Min` `Max` `Range` `Positive` `NonNegative`
`Finite`

**Booleans** — `Required` `Default` `MustBe`

**Times** — `Required` `Default` `UTC` `After` `Before` `Past` `Future`

**Durations** — `Required` `Default` `Min` `Max` `Positive`

**Lists** (`Strings`, `Ints`) — `Required` `CSV` `MinItems` `MaxItems`
`NonEmpty` `Unique` `OneOf` *(strings)* `Each`

`Range(min, max)` is `Min` and `Max` together, and on a string it bounds the
length. That is the ambiguity a tag could not resolve — `min=1` meaning a value
on an integer and a length on a string — and here they are simply different
methods on different types.

`Lowercase` refuses a value that is not already lower case; `ToLower` folds it
instead. Pick the one that matches what the API promises.

## Every entry of a list

`Each` gives the entry builder for the list's own element type, so everything a
string or an integer can be asked is available:

```go
b.Strings(&in.Tags).Each(func(tag *tork.StringParam) {
	tag.Trim().ToLower().MaxLen(20).Slug()
})

b.Ints(&in.Sizes).Each(func(size *tork.IntParam) {
	size.Range(1, 10)
})
```

Transforms apply to every entry, and a failure names the list, since an entry
has no name of its own:

```json
{"field": "query.tags", "issue": "too_long", "message": "every value in tags must be at most 20 characters."}
```

## Rules of your own

`Check` takes an issue, a predicate that completes the sentence, and a function
of the field's own type:

```go
b.String(&in.Name).Check("reserved_word", "must not be a reserved word",
	func(name string) bool { return !reserved[name] })

b.Int(&in.Count).Check("must_be_even", "must be an even number",
	func(count int64) bool { return count%2 == 0 })

b.Time(&in.When).Check("not_monday", "must not be a Monday",
	func(when time.Time) bool { return when.Weekday() != time.Monday })
```

```json
{"field": "name", "issue": "reserved_word", "message": "name must not be a reserved word."}
```

There is nothing to type-assert: `Check` on a string hands you a `string`, on an
integer an `int64`, on a list the whole slice. The issue is yours to name.

## Nested documents

A body is a document, not a flat list of fields. A struct whose own type has
rules is checked through, and so is every entry of a list of them — the names
build up as the walk goes:

```go
type Address struct {
	Line1   string `json:"line1"`
	ZipCode string `json:"zipCode"`
}

var addressRules = tork.DefineBody(func(b *tork.BodyRules, in *Address) {
	b.String(&in.Line1).Required().Trim()
	b.String(&in.ZipCode).Required().Trim().ToUpper()
})

type CheckoutBody struct {
	tork.JSONBody
	Billing  Address    `json:"billing"`
	Shipping *Address   `json:"shipping"`
	Items    []LineItem `json:"items"`
}
```

```json
{
  "details": [
    {"field": "billing.zipCode",   "issue": "field_required",    "message": "zipCode is required."},
    {"field": "items[1].sku",      "issue": "pattern_mismatch",  "message": "sku must begin with sku_."},
    {"field": "items[1].quantity", "issue": "minimum_not_met",   "message": "quantity must be greater than zero."}
  ]
}
```

The rules travel with the type, so `Address` is declared once and checked
wherever it appears. A nested struct nobody sent, or a nil pointer, has nothing
inside it to complain about. A type that can contain itself is walked once, not
forever.

An embedded struct's fields are written at the top level by `encoding/json`, so
that is where a complaint about them is reported — no prefix.

## Two fields at once

`Check` on the declaration itself sees the whole decoded body, which is the only
way to say something about two fields together:

```go
var bookingBody = tork.DefineBody(func(b *tork.BodyRules, in *BookingBody) {
	b.Time(&in.From).Required()
	b.Time(&in.Until).Required()
}).Check(func(in *BookingBody) []tork.FieldError {
	if !in.Until.After(in.From) {
		return []tork.FieldError{{
			Field:   "until",
			Issue:   "before_start",
			Message: "until must be after from.",
		}}
	}
	return nil
})
```

It runs after every field has been judged on its own, and **only when they all
passed** — a check comparing two fields has nothing useful to say about a
document already known to be wrong. Fields it names are relative to the body, so
a body nested inside another still reports the path a client would recognise
(`booking.until`).

## When a rule runs

Only against a value the client actually sent.

- **Absent** — nothing to judge, so the rules are skipped. Use `Required` to
  insist on a value.
- **A default** — the author's own value, so it is not put through the rules the
  client's values are judged by.
- **An `Optional` that is unset or explicitly null** — carries no value, so
  rules and transforms are both skipped. `Required` still applies.
- **A body field holding its zero value** — a body cannot tell a field that was
  left out from one sent as its zero value, so the zero value counts as absent.

That last rule has one consequence worth stating outright: on a body field,
`b.Int(&in.Quantity).Positive()` **will not reject a zero**, because zero is
what absent looks like. Write `Required().Positive()`, or make the field
[`Optional[int]`](/docs/2.fundamentals/8.optional-values), where absent and zero
are different things.

## What a broken rule answers

The same envelope everything else uses, and **every** broken rule is reported —
along with every field that failed to bind at all:

```json
{
  "code": "VALIDATION_ERROR",
  "message": "Validation failed for 3 fields.",
  "status": 400,
  "timestamp": "2026-07-23T16:37:23Z",
  "path": "/items",
  "details": [
    {"field": "query.page",   "issue": "minimum_not_met",  "message": "page must be at least 1."},
    {"field": "query.limit",  "issue": "maximum_exceeded", "message": "limit must be at most 100."},
    {"field": "query.search", "issue": "too_long",         "message": "search must be at most 200 characters."}
  ]
}
```

Parameters are named by where they came from; body fields keep their dotted
path. See [Errors](/docs/2.fundamentals/11.errors).

## Issues

`Issue` is the half a client acts on, so each is a constant. `Message` is prose
and may be reworded at any time.

| Rule | Issue |
|---|---|
| `Required`, `NotBlank` | `field_required` |
| `Min`, `MinLen` on a number or duration, `Positive`, `NonNegative` | `minimum_not_met` |
| `Max` on a number or duration, `Negative`, `NonPositive` | `maximum_exceeded` |
| `MinLen`, `Len` on a string | `too_short` |
| `MaxLen` on a string | `too_long` |
| `MinItems` / `MaxItems` | `too_few_items` / `too_many_items` |
| `Unique` | `duplicate_items` |
| `OneOf`, `MustBe` | `not_in_set` |
| `Pattern`, `Contains`, `HasPrefix`, `HasSuffix` | `pattern_mismatch` |
| `MultipleOf` | `not_multiple_of` |
| `After`, `Future` | `too_early` |
| `Before`, `Past` | `too_late` |
| `Email` / `UUID` / `URL` | `invalid_email` / `invalid_uuid` / `invalid_url` |
| every other format | `invalid_format` |
| `Finite` | `invalid_number` |

They are exported as `tork.IssueMinimumNotMet` and so on, beside the ones a
value that could not be read at all carries — `invalid_integer`,
`invalid_type`, and the rest. The two groups are deliberately separate: one
means the client sent nonsense, the other that it sent something sensible that
is not allowed here.

## Validating in the handler

A check that needs the database, or another request, belongs in the handler and
answers with the same envelope:

```go
if taken, _ := store.NameTaken(ctx, body.Name); taken {
	return Item{}, tork.ValidationFailed(tork.FieldError{
		Field:   "name",
		Issue:   "already_taken",
		Message: "That name is already in use.",
	})
}
```

## The formats, precisely

`Email` is deliberately loose: something, an `@`, and something with a dot in
it. Anything stricter rejects addresses that are perfectly valid, and the only
real proof an address works is sending to it.

`UUID` accepts the canonical 8-4-4-4-12 hex form in either case. `URL` requires
a scheme and a host. `Hostname` follows RFC 1123 — labels of letters, digits,
and hyphens, none beginning or ending with one, and a trailing dot allowed.
`CreditCard` applies the Luhn check, which says a number is well formed, not
that the card exists. `Alpha`, `Alphanumeric`, and `Numeric` are Unicode-aware,
so a name in any script passes.

# Query parameters

A field tagged `query` reads the URL's query string:

```go
type ListItemsInput struct {
	Page   int      `query:"page"   default:"1"`
	Limit  int      `query:"limit"  default:"20"`
	Search string   `query:"search"`
	Sort   []string `query:"sort"`
	Active *bool    `query:"active"`
}

func listItems(ctx context.Context, in ListItemsInput) ([]ItemResponse, error)
```

```
GET /items?page=3&search=boots&sort=name&sort=-created
```

## Names

An empty tag derives the name from the field, lower-camel-cased. An explicit
name always wins:

```go
type ListItemsInput struct {
	Page     int    `query:""`   // ?page=
	PageSize int    `query:""`   // ?pageSize=
	ItemID   string `query:""`   // ?itemId=
	Search   string `query:"q"`  // ?q=
}
```

The rule splits the field name into words at its case boundaries, lowercases
the first, and title-cases the rest: `PageSize` → `pageSize`, `ItemID` →
`itemId`, `HTTPServer` → `httpServer`. The initialism is deliberately not
preserved, because a name that never varies is worth more than one that matches
a given reader's preference — it is a wire contract, not a Go identifier.

Two fields reading the same name is a build error:

```
items/handler.go:8: field Other reads query.page, which field Page already reads
```

## Absent, empty, and defaults

A parameter that is absent takes its `default`. A parameter sent with an empty
value counts as absent, so `?page=` and leaving `page` out mean the same thing:

```go
Page int `query:"page" default:"1"`
```

```
GET /items          ->  Page == 1
GET /items?page=    ->  Page == 1
GET /items?page=3   ->  Page == 3
```

With no default, an absent parameter leaves the field at its zero value.

The default is parsed when the application builds, so one that does not fit its
field is a build error rather than a value that silently never applies:

```
items/handler.go:8: field Page has an invalid default "lots":
    it must be a whole number
```

If you need to tell "sent empty" from "not sent", use a pointer or an
[`Optional[T]`](/docs/2.fundamentals/8.optional-values), where the distinction
is the point.

## Repeated values

A slice field collects every value the client sent:

```go
Sort []string `query:"sort"`
```

```
?sort=name&sort=-created   ->  ["name", "-created"]
```

A scalar field sent more than once keeps the last value, since that is what a
client overriding an earlier one would expect.

## Comma-separated lists

The `,csv` modifier splits one value into many instead:

```go
Tags []string `query:"tags,csv"`
```

```
?tags=new,sale   ->  ["new", "sale"]
```

`csv` on a field that is not a slice is a build error, and it is the only
modifier there is — anything else is refused by name.

## Types

Every source binds the same set of types:

| Type | Read as |
|---|---|
| `string` | as sent |
| `bool` | `true`/`false`, `1`/`0`, `yes`/`no`, `on`/`off`, any case |
| `int`, `int8` … `int64` | a whole number, parsed within the field's own width |
| `uint`, `uint8` … `uint64` | the same, and not negative |
| `float32`, `float64` | a number |
| `time.Time` | RFC 3339 — `2026-07-23T16:37:23Z` |
| `time.Duration` | Go's own form — `1h30m`, `500ms` |
| any `encoding.TextUnmarshaler` | whatever the type accepts |
| `*T` | the above, left nil when absent |
| `[]T` | one element per value |
| `Optional[T]` | the above, remembering whether it was sent |

`yes`, `no`, `on`, and `off` are accepted for booleans because HTML forms send
them and a checkbox is the commonest boolean there is.

Parsing within the field's width means an out-of-range value is refused rather
than silently truncated: `?small=999` into an `int8` is an error, not `-25`.

## Your own types

Anything implementing `encoding.TextUnmarshaler` binds without the framework
knowing about it:

```go
type SKU struct{ code string }

func (s *SKU) UnmarshalText(text []byte) error {
	if !strings.HasPrefix(string(text), "sku_") {
		return fmt.Errorf("a SKU begins with sku_")
	}
	s.code = string(text)
	return nil
}

type FindInput struct {
	SKU SKU `query:"sku"`
}
```

A value the type refuses is reported as `invalid_format`.

## Sharing parameters

An embedded struct is walked, so a set of parameters used by several endpoints
can be declared once:

```go
type Pagination struct {
	Page  int `query:"page"  default:"1"`
	Limit int `query:"limit" default:"20"`
}

type ListItemsInput struct {
	Pagination
	Search string `query:"search"`
}
```

An exported field of an input struct with no binding tag is a build error — a
misspelled tag would otherwise leave the field silently empty forever.
Unexported fields are ignored, since nothing could be written to them.

## When a value cannot be read

Every field is attempted before the request is refused, so one response reports
everything wrong:

```
GET /items?page=abc&limit=xyz
```

```json
{
  "code": "VALIDATION_ERROR",
  "message": "Validation failed for 2 fields.",
  "status": 400,
  "timestamp": "2026-07-23T16:37:23Z",
  "path": "/items",
  "details": [
    {"field": "query.page",  "issue": "invalid_integer", "message": "page must be a whole number."},
    {"field": "query.limit", "issue": "invalid_integer", "message": "limit must be a whole number."}
  ]
}
```

Parameters are named by where they came from, so `query.page` can never be
confused with a body field called `page`.

Rules about a value — a range, a length, a pattern — are not written as tags at
all. They are declared in code, where the compiler can check them; see
[Typed declarations](/docs/2.fundamentals/9.typed-declarations) and
[Validation](/docs/2.fundamentals/10.validation).

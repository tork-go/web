# Optional values

JSON has three answers to "what did the client say about this field" and Go has
two. A `*string` that is nil cannot say whether the field was left out or sent
as `null`, and for a PATCH those mean opposite things: leave it alone, or clear
it.

`Optional[T]` is the third answer:

```go
type PatchItemBody struct {
	tork.JSONBody
	Name  tork.Optional[string] `json:"name,omitzero"`
	Price tork.Optional[int64]  `json:"price,omitzero"`
}

func patchItem(ctx context.Context, in PatchItemInput) (ItemResponse, error) {
	item, err := store.Find(ctx, in.ItemID)
	if err != nil {
		return ItemResponse{}, err
	}

	if name, ok := in.Body.Name.Get(); ok {
		item.Name = name          // sent, and not null
	}
	if in.Body.Price.IsNull() {
		item.Price = nil          // sent as null
	}
	// a field that was left out is neither, and item keeps what it had

	return store.Save(ctx, item)
}
```

## Reading one

| Method | Answers |
|---|---|
| `Get() (T, bool)` | the value, and whether there is a usable one |
| `Or(fallback T) T` | the value, or the fallback |
| `IsSet() bool` | whether the field was mentioned at all, `null` included |
| `IsNull() bool` | whether it was mentioned and was `null` |
| `IsZero() bool` | whether it was left out |

`Get` reports `false` for both an absent field and an explicit null, which is
what a caller that only wants a usable value should ask. `IsSet` and `IsNull`
are how you tell those two apart.

## Building one

```go
tork.Some("Boots")   // a value
tork.Null[string]()  // an explicit null
tork.Optional[string]{}  // absent â€” the zero value
```

The fields are unexported on purpose. An `Optional` built as a literal could
otherwise carry a value it does not consider set, and there is no reading of
that anyone meant.

## In responses

The zero value is the absent one, and `IsZero` says so, which is exactly what
`omitzero` asks a type. So the same type works on the way out:

```go
type ItemResponse struct {
	ID    string                `json:"id"`
	Name  tork.Optional[string] `json:"name,omitzero"`
	Price tork.Optional[int64]  `json:"price,omitzero"`
}
```

| Value | Encodes as |
|---|---|
| `Optional[string]{}` | *(the field is left out)* |
| `tork.Some("Boots")` | `"name": "Boots"` |
| `tork.Null[string]()` | `"name": null` |

Without `omitzero` the field is always written, and an absent `Optional` has to
be something â€” `null` is the only honest answer. Tag it `omitzero` unless you
want that.

`omitzero` is a Go 1.24 feature, and Tork Web's baseline is 1.25, so it is
always available.

## In query strings

A query string has no `null`, so an `Optional` there tells present from absent:

```go
type SearchInput struct {
	Search tork.Optional[string] `query:"search"`
	Page   tork.Optional[int]    `query:"page"`
}

if search, ok := in.Search.Get(); ok {
	// ?search= was sent with something in it
}

limit := in.Page.Or(1)
```

This is also how to tell "sent empty" from "not sent" â€” with one caveat worth
knowing. Tork treats an empty value as absent everywhere, so `?search=` is *not*
set. `Optional` distinguishes absent from present-with-a-value, not absent from
present-and-empty.

The same works for headers, cookies, and form fields.

## Errors

A value that cannot be read into the `Optional`'s element type is reported like
any other, against the field:

```json
{"field": "query.page", "issue": "invalid_integer", "message": "page must be a whole number."}
```

# Dependencies

A handler takes the services it needs as ordinary parameters, and the framework
supplies them:

```go
func readItem(
	ctx context.Context,
	in  ReadItemInput,
	svc *Service,             // built once, shared by every request
	user auth.Principal,      // built per request, from a Depends
) (ItemResponse, error)
```

Nothing is fetched from a container and no global is reached for. The graph of
what builds what is declared on the routers a feature package already returns,
checked when the application builds, and constructed once — so a request pays
for none of it.

## Providers

`tork.Provide` registers constructors. Each is a function that returns the
value it provides, in one of three shapes:

```go
tork.Provide(
	NewRepository,   // func(deps...) T
	NewService,      // func(deps...) (T, error)
	OpenDatabase,    // func(deps...) (T, tork.Cleanup, error)
)
```

A constructor's own parameters are resolved from the graph, so constructors
compose: `NewService` taking a `*Repository` is wired to whatever provides one,
without either naming the other's package twice. Everything `Provide` builds is
a **singleton** — constructed once, when the application builds, and shared by
every request — because a service is a thing the program has, not a thing a
request makes.

The three return shapes differ only in what a constructor says beside the
value. A bare `T` cannot fail; `(T, error)` can; `(T, tork.Cleanup, error)`
returns a `Cleanup` to run at shutdown as well.

Providers are visible application-wide. A repository declared under `/items`
can be injected into a handler under `/users`, because a service graph is a
property of the program, not of a URL prefix. Two providers for one type is a
build error naming both.

### A value already built

`tork.ProvideValue` registers a value you already have, rather than a recipe
for one:

```go
db  := openDatabase()
cfg := config.Load()

app := tork.New(
	tork.ProvideValue(db),
	tork.ProvideValue(cfg),
)
```

It exists so a value in hand does not have to be wrapped in a constructor that
only returns it. The type it provides is the value's own dynamic type, so a
value passed as an interface provides that interface's concrete type — see
[interfaces](#interfaces-and-the-named-type) for when the interface is what a
handler asks for.

## Depends

`tork.Depends` registers a function that runs before the handler, once per
request. What it returns decides what it is:

```go
tork.Depends(auth.RequireToken)   // (Principal, error): authenticates and provides
tork.Depends(audit.Record)        // error: runs for its effect
```

A function returning a value makes that value **request-scoped**: any handler
or later dependency under it can ask for it by type, and it is built once per
request and shared by everyone in that request who reads it. A function
returning only an `error` is a **guard**: it runs for its effect and
contributes no value, but it can refuse the request, and its error travels
through the [error envelope](/docs/2.fundamentals/12.errors) exactly as a
handler's would. When a guard refuses, the handler never runs.

A dependency takes exactly what a handler takes — a context, the request, an
input struct, an injected service, or a value an outer dependency produced — so
it can read a header or a token through its own input struct rather than a
second mechanism:

```go
type TokenInput struct {
	Token string `header:"Authorization"`
}

func RequireToken(ctx context.Context, in TokenInput) (Principal, error) {
	p, err := verify(in.Token)
	if err != nil {
		return Principal{}, tork.Unauthorized("invalid or missing token")
	}
	return p, nil
}
```

Dependencies declared further out run first — the application's, then a
router's, then a route's — because an inner one may rely on what an outer one
established. A value dependency is registered only after its own parameters are
compiled, so it can draw on what ran before it but not on itself.

## The three scopes

| Scope | Declared with | Built | Shared |
|---|---|---|---|
| Singleton | `Provide`, `ProvideValue` | once, at startup | for the life of the application |
| Request-scoped | `Depends` | once per request | within that request |
| Transient | `Provide(Transient(…))` | at each injection | never |

Most services are one of the first two. `tork.Transient` is the rare third:

```go
tork.Provide(NewRepository, tork.Transient(NewScratchBuffer))
```

It builds a fresh value at every place it is injected, for a value that must
not be aliased — a builder two callers would corrupt, a buffer each use needs
its own of. A transient is built from the singletons in the graph, not from
anything request-scoped, and cannot register a `Cleanup`: its lifetime is the
injection, not the application, so there is no shutdown for one to run at.

## Interfaces and the named type

The graph is keyed by type, never by a string name. Two implementations of one
interface — a primary and a replica database, say — are two distinct **named
types**, not one type under two labels:

```go
type PrimaryDB struct{ *sql.DB }
type ReplicaDB struct{ *sql.DB }

tork.Provide(NewPrimaryDB, NewReplicaDB)

func readItem(ctx context.Context, in ReadItemInput, db ReplicaDB) (ItemResponse, error)
```

A wrapper type is checked by the compiler; a string name is not. This is the
only way to resolve an interface with more than one implementation, and it is
deliberately the only one.

## Cleanup

A provider or a dependency that acquires something releases it with a
`tork.Cleanup`, which is `func(context.Context) error`. Cleanups run in reverse
of the order they were registered, so a thing is torn down before whatever it
was built on top of.

A **request-scoped** cleanup runs after the response, whatever became of it —
so a dependency that opened a transaction has it closed even when a later
dependency, or the handler, failed. A **singleton** cleanup runs when the
application shuts down: `Serve` and `ServeListener` run them for you after the
server stops, and `App.Close` runs them for an application served some other
way.

Singletons are constructed **eagerly**, when `Handler` builds, not on first
use. A database that will not open should fail the boot, not the first request
that happens to need it; and a constructor that fails unwinds what was already
built, so a half-constructed graph never leaks what it did acquire.

## Build errors

Everything about the graph is checked when the application builds, and every
mistake is reported together with the rest, each naming the line it was written
on.

| Situation | Message |
|---|---|
| A provider is not a function | `a provider is a T, not a function` |
| A provider's results are the wrong shape | `a provider returns N values; a provider returns T, (T, error), or (T, tork.Cleanup, error)` |
| A function returning only `error` is given to `Provide` | `a function returning only an error is a tork.Depends guard, not a provider` |
| Two providers for one type | `a provider for T is already declared at file:line` |
| A constructor parameter nothing provides | `the provider for T needs U, which nothing provides; declare it with tork.Provide` |
| A singleton depends on the request | `the provider for T takes U, which exists only per request; a singleton cannot depend on the request` |
| The graph has a cycle | `providers form a cycle: A -> B -> A` |
| A transient registers a cleanup | `a transient provider cannot register a tork.Cleanup; its lifetime is the injection, not the application` |
| A handler or dependency parameter nothing provides | `nothing provides T; take it from a tork.Provide or a tork.Depends` |

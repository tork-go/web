# Handlers

A handler is a function. There is no interface to satisfy and no framework type
you are obliged to take:

```go
func readItem(ctx context.Context, in ReadItemInput) (ItemResponse, error) {
	return ItemResponse{ID: in.ItemID}, nil
}
```

Its signature is read once, when the application builds, and compiled into the
closures that serve it. Nothing about the signature is inspected again while a
request is in flight.

## Parameters

Each parameter is classified on its own, in this order:

1. `context.Context` — the request's context.
2. `*tork.Request` — the raw request and response. See
   [the escape hatch](#the-escape-hatch) below.
3. A struct declared with
   [`tork.DefineInput`](/docs/2.fundamentals/9.typed-declarations), or carrying
   `path`, `query`, `header`, `cookie`, `form`, or `body` tags — a request
   input, decoded from the request.
4. A struct embedding `tork.JSONBody` — a request body.
5. Anything else — a dependency, which is not built yet and is currently a
   build error.

Order does not matter, and you may take as many input structs as you like:

```go
func readItem(
	ctx context.Context,
	path ReadItemPath,
	query ReadItemQuery,
) (ItemResponse, error)
```

A handler may take no parameters at all:

```go
func health() (Health, error) {
	return Health{Status: "ok"}, nil
}
```

## Results

A handler returns `(T, error)` or `error` alone. The error is last because that
is where Go puts it, and it is required because a handler that cannot fail is
rare enough that letting it say so would cost every other handler a second shape
to remember.

```go
func readItem(ctx context.Context, in ReadItemInput) (ItemResponse, error)
func deleteItem(ctx context.Context, in DeleteItemInput) error
```

`T` is serialized as JSON with status 200. A handler returning only an error
answers **204 No Content** when it succeeds, because it has nothing to say.

Anything else is a build error:

```
items/handler.go:12: GET /items: returns (items.ItemResponse, string);
    the second result must be error
```

That is the whole contract for the common case. When it is not — a different
status, an extra header, a file download, a redirect, a body written as it
goes — `T` can be `tork.Response[T]`, `tork.RawResponse`, `tork.FileResponse`,
`tork.Stream`, or `tork.Redirect` instead, each deciding its own status and
headers rather than leaving 200 assumed. See
[Responses](/docs/2.fundamentals/11.responses).

## Errors

Returning an error is how a handler refuses a request:

```go
func readItem(ctx context.Context, in ReadItemInput) (ItemResponse, error) {
	item, err := store.Find(ctx, in.ItemID)
	if err != nil {
		return ItemResponse{}, err
	}
	if item == nil {
		return ItemResponse{}, tork.NotFound("Item", in.ItemID)
	}
	return toResponse(item), nil
}
```

An error that does not say what it should become answers 500, with its message
logged rather than served. See [Errors](/docs/2.fundamentals/12.errors) for the
envelope, for making your own error types answer for themselves, and for
mapping a domain error to a status in one place.

## Operation IDs

Every route gets an operation ID, which is the stable name a generated client
will give it. Left unset, it is derived from the handler's own function name:

```go
router.GET("/{item_id}", readItem)          // items.readItem
router.GET("/{item_id}", readItem, tork.OperationID("items.read"))
```

The derived name is the package and function, which is stable as long as the
function is. A function literal has no name worth using, so a closure handler
falls back to the route: `get.items.item_id`.

Two routes claiming the same ID within one API version is a build error.

## Panics

A handler that panics answers 500 rather than taking the connection with it. The
panic value and a stack are logged; nothing about it reaches the client.

`http.ErrAbortHandler` is passed through untouched, since that is `net/http`'s
own signal for "stop, silently".

## The escape hatch

`*tork.Request` gives you the underlying request and response for whatever the
framework does not model — streaming a body, hijacking a connection, reading a
header no struct wants to own:

```go
func upload(ctx context.Context, r *tork.Request) (Result, error) {
	raw := r.HTTP()          // *http.Request
	w := r.Writer()          // http.ResponseWriter
	id := r.PathValue("id")  // a wildcard, unnamed by any struct
}
```

Taking one is a deliberate step outside the typed path: nothing read through it
will appear in the OpenAPI document, because nothing declared it. Writing to
`Writer()` takes over the response, and whatever the handler returns afterwards
is written on top of what has already been sent.

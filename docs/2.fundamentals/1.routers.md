# Routers

A router is not a multiplexer. Nothing is dispatched by one and no path is
matched; it holds declarations, and the application folds them into finished
routes when it builds. That is what lets a feature package return a `*Router`
naming its own prefix and tags without knowing where it will end up.

```go
package items

func Router() *tork.Router {
	router := tork.NewRouter(
		tork.Prefix("/items"),
		tork.Tags("items"),
	)

	router.GET("/", listItems)
	router.GET("/{item_id}", readItem)
	router.PUT("/{item_id}", updateItem)

	return router
}
```

```go
app := tork.New(tork.Title("Example API"))
v1 := app.Version("v1", tork.Prefix("/api/v1"))
v1.Include(items.Router())
```

`NewRouter` cannot fail, and that is deliberate: the prefix a router
contributes has to land after whatever prefix it is mounted under, which is not
known at the point the router is written. Its options are therefore applied
when it is mounted, and any mistake in them is reported when the application
builds.

## Declaring routes

```go
router.GET(path, handler, opts...)
router.POST(path, handler, opts...)
router.PUT(path, handler, opts...)
router.PATCH(path, handler, opts...)
router.DELETE(path, handler, opts...)
router.OPTIONS(path, handler, opts...)
router.Handle(method, path, handler, opts...)
```

`Handle` takes any method for the ones the named helpers do not cover; it
upper-cases what you give it. The same methods exist on `*App` and declare
routes at the root.

A path must begin with a slash and must not end with one. A path of `"/"` means
the router's prefix itself, which is the idiom for the collection a router is
mounted at:

```go
router := tork.NewRouter(tork.Prefix("/items"))
router.GET("/", listItems)          // GET /items
router.GET("/{item_id}", readItem)  // GET /items/{item_id}
```

The trailing slash matters more than tidiness. `net/http.ServeMux` reads one as
a subtree match, so `/items/` would match every path beginning with `/items`
rather than the route declared for it. Tork never emits one.

## Mounting

`Include` mounts a router underneath another. Options given at the include site
land between the parent's and the child's:

```go
v1.Include(
	admin.Router(),          // itself prefixed /settings
	tork.Prefix("/admin"),
	tork.Tags("admin"),
)
// GET /api/v1/admin/settings
```

Routes come out in the order they were written, whether they were declared
directly on a router or came from something included into it.

## Versions

`app.Version` declares an API version: a router whose routes will get their own
OpenAPI document once that subsystem lands.

```go
v1 := app.Version("v1", tork.Prefix("/api/v1"))
v2 := app.Version("v2", tork.Prefix("/api/v2"))
```

The name is also the scope an operation ID has to be unique within, so two
versions may both have an `items.read`.

## What is inherited

| Option | Inherited | Notes |
|---|---|---|
| `Prefix` | yes | concatenated, outermost first |
| `Tags` | yes | accumulated and deduplicated, in declaration order |
| `Use` | yes | outermost runs first |
| `Deprecated` | yes | marks every operation underneath |
| `Responds` | yes | merged by status; a route redeclaring one replaces it |
| `Throws` | yes | accumulated; the same type twice is not a collision |
| `Summary` | no | belongs to one operation |
| `Description` | no | belongs to one operation, or to the API |
| `OperationID` | no | must be unique |

Documentation is not inherited on purpose. A summary that applied to every
route under a router would be wrong on all but the one it was written for.

Two routers included into the same parent never see each other's tags, so this
is safe:

```go
app := tork.New(tork.Tags("api"))
app.Include(items.Router())   // tags: api, items
app.Include(users.Router())   // tags: api, users
```

## Where options may appear

Every option knows which of the three sites accepts it, and using one in the
wrong place is a build error naming the line you wrote it on:

```
items/router.go:24: tork.Summary is not valid on a router
```

- **Application** (`tork.New`): `Title`, `Description`, `Clock`, `Logger`,
  `OnError`, `WriteErrorsWith`, `MaxBodyBytes`, `StrictBodies` — plus every
  router option, since the application is also the root of the tree.
- **Router** (`NewRouter`, `Version`, `Include`): `Prefix`, `Tags`, `Use`,
  `Deprecated`, `Responds`, `Throws`.
- **Route** (`GET`, `POST`, …): `Summary`, `Description`, `OperationID`,
  `Deprecated`, `Tags`, `Use`, `Responds`, `Throws`.

`Responds` and `Throws` document a response beyond what a handler's own
result type already says — see
[Responses](/docs/2.fundamentals/11.responses).

## Collisions

Two collisions are refused when the application builds:

```
items/router.go:31: GET /items is already declared at items/router.go:24
items/router.go:31: operation ID "items.read" is already used by GET /items at
    items/router.go:24; set tork.OperationID on one of them
```

A third is caught by `ServeMux` and reported the same way: two paths that differ
only in the names of their wildcards, such as `/items/{item_id}` and
`/items/{id}`. They are the same path as far as matching is concerned.

# Path parameters

A wildcard in a route pattern binds to a field tagged `path`:

```go
type ReadItemInput struct {
	ItemID string `path:"item_id"`
}

func readItem(ctx context.Context, in ReadItemInput) (ItemResponse, error) {
	return store.Find(ctx, in.ItemID)
}

router.GET("/{item_id}", readItem)
```

The `{item_id}` syntax is `net/http.ServeMux`'s own — Tork adds no router of
its own — so a pattern you already know how to write is the pattern you write
here.

## Checked against the route

A `path` field naming a wildcard the route does not have is a build error. It
could only ever bind empty, so there is no reading of it that works:

```
items/router.go:24: GET /items: parameter 1: field ItemID reads the path
    parameter "item_id", but GET /items has no {item_id} in it
```

The reverse is allowed. A wildcard nothing reads is fine, because something
further up may be what consumes it:

```go
app.GET("/tenants/{tenant}/items/{item_id}", readItem)
```

Here `readItem` reads only `item_id`; `tenant` is there for the middleware, or
for a [dependency](/docs/2.fundamentals/14.dependencies) that reads it through
its own input struct to resolve the tenant.

## Names

An empty tag derives the name from the field, lower-camel-cased:

```go
type ReadItemInput struct {
	ItemID string `path:""`   // reads {itemId}
}

router.GET("/{itemId}", readItem)
```

Whether you write `{item_id}` and tag it explicitly or write `{itemId}` and let
it derive is a house-style choice; the framework has no preference. Explicit
always wins over derived.

## Multi-segment wildcards

`ServeMux` spells a wildcard that swallows the rest of the path `{name...}`. The
name is the same either way:

```go
type FileInput struct {
	Rest string `path:"rest"`
}

app.GET("/files/{rest...}", serveFile)
// GET /files/a/b/c.txt  ->  Rest == "a/b/c.txt"
```

## Types

A path parameter is a string on the wire, and binds into any type that can be
read from one — see [Query parameters](/docs/2.fundamentals/4.query-parameters#types)
for the full list, which is shared by all five sources.

```go
type ReadItemInput struct {
	ItemID  int       `path:"item_id"`
	Version time.Time `path:"version"`
}
```

A value that does not fit is reported against the field:

```json
{
  "field": "path.item_id",
  "issue": "invalid_integer",
  "message": "item_id must be a whole number."
}
```

## A note on matching

Because Tork registers each route as an exact pattern, a path that no route
matches answers 404 with `ROUTE_NOT_FOUND` rather than falling into some other
route. That includes the root: `app.GET("/", root)` matches `/` and nothing
else, which is not what a bare `"/"` pattern does in `ServeMux` on its own.

Two routes whose paths differ only in the names of their wildcards —
`/items/{item_id}` and `/items/{id}` — are the same path as far as matching
goes, and are refused when the application builds.

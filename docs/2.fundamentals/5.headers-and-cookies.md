# Headers and cookies

```go
type TracedInput struct {
	TraceID   string `header:"X-Trace-ID"`
	UserAgent string `header:"User-Agent"`
	Session   string `cookie:"session"`
	Theme     string `cookie:"theme"`
}
```

Both bind exactly like [query parameters](/docs/2.fundamentals/4.query-parameters):
the same types, the same `default` tag, the same treatment of an empty value as
an absent one.

## Header names are always explicit

`header:""` is a build error:

```
auth/input.go:8: field TraceID has an empty header tag; a header name cannot be
    derived from a Go identifier, so write it out
```

There is no derivation that gets from `TraceID` to `X-Trace-ID`, and one that
guessed would be wrong more often than right. Cookies and the other three
sources do derive from the field name; headers do not.

Lookup is case-insensitive, as HTTP requires, so `header:"x-trace-id"` and
`header:"X-Trace-ID"` read the same header.

## Repeated headers

A header sent more than once fills a slice, the same as a repeated query
parameter:

```go
type ForwardedInput struct {
	Forwarded []string `header:"X-Forwarded-For"`
}
```

## Cookies

A cookie that was not sent leaves its field at the zero value. There is no
distinction between "no such cookie" and "a cookie with an empty value", so use
an [`Optional[T]`](/docs/2.fundamentals/8.optional-values) if that matters:

```go
type SessionInput struct {
	Session tork.Optional[string] `cookie:"session"`
}

if session, ok := in.Session.Get(); ok {
	// the cookie was sent
}
```

Setting a cookie on the way out goes through the raw response for now, since
typed response wrappers are not built yet:

```go
func login(ctx context.Context, r *tork.Request, in LoginInput) (Session, error) {
	http.SetCookie(r.Writer(), &http.Cookie{
		Name:     "session",
		Value:    token,
		HttpOnly: true,
		Path:     "/",
	})
	return Session{Token: token}, nil
}
```

Headers set this way are written before the response body, so this is safe: the
handler returns normally and the framework writes the status and body
afterwards.

## Errors

A header or cookie that cannot be read into its field is reported like any other
parameter, named by its source:

```json
{
  "details": [
    {"field": "header.X-Retry-After", "issue": "invalid_integer", "message": "X-Retry-After must be a whole number."},
    {"field": "cookie.visits",        "issue": "invalid_integer", "message": "visits must be a whole number."}
  ]
}
```

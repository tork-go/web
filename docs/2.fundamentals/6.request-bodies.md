# Request bodies

There are two ways to declare a body, and which to use depends on whether the
handler needs anything besides it.

## Inside an input struct

The form to reach for whenever the handler also has path, query, header, or
cookie parameters:

```go
type UpdateItemBody struct {
	Name  string `json:"name"`
	Price int64  `json:"price"`
}

type UpdateItemInput struct {
	ItemID string         `path:"item_id"`
	DryRun bool           `query:"dryRun"`
	Token  string         `header:"X-Token"`
	Body   UpdateItemBody `body:"json"`
}

func updateItem(ctx context.Context, in UpdateItemInput) (ItemResponse, error) {
	return store.Update(ctx, in.ItemID, in.Body)
}
```

The body struct is a plain struct with `json` tags. It knows nothing about the
framework.

## Taken directly

A struct embedding `tork.JSONBody` *is* a body, so a handler that needs nothing
else takes it as a parameter with no wrapper:

```go
type CreateItemBody struct {
	tork.JSONBody
	Name  string `json:"name"`
	Price int64  `json:"price"`
}

func createItem(ctx context.Context, body CreateItemBody) (ItemResponse, error) {
	return store.Create(ctx, body)
}
```

The marker is what makes this possible. Without it there would be no way to
tell a body from an injected dependency: both are plain structs, and a body's
`json` tags say what its fields are called, not where the value comes from.

A struct that embeds the marker **and** carries parameter tags is a build error.
It would have two answers to what it is, so it gets neither — that is what the
wrapper form is for.

## One body per handler

A request has one body, however the handler's inputs are divided up. Declaring
two is a build error, and so is mixing a body with
[form fields](/docs/2.fundamentals/7.forms-and-uploads), since both consume the
request:

```
items/handler.go:14: items.SecondBody is a second request body;
    items.FirstBody already is one, and a request has one body
```

## Content types

The body must be `application/json`, or a structured suffix like
`application/merge-patch+json`. Parameters are ignored, so
`application/json; charset=utf-8` is fine.

A missing `Content-Type` is accepted: plenty of clients omit it, and the body
still has to parse as JSON to get any further. Anything else answers **415**:

```json
{
  "code": "UNSUPPORTED_MEDIA_TYPE",
  "message": "This endpoint reads application/json, not \"text/plain\".",
  "status": 415,
  "timestamp": "2026-07-23T16:37:23Z",
  "path": "/items"
}
```

## Size

Bodies are limited to 1 MiB by default. Exceeding it answers **413** without
reading the rest:

```go
app := tork.New(tork.MaxBodyBytes(5 << 20))
```

An endpoint that takes uploads should say so rather than having every other
endpoint pay for the possibility.

## Unknown fields

A field the body carries and the struct does not is ignored by default. That is
how an API is extended without breaking the clients that have not caught up.

For an API where an unrecognised field more likely means a misspelled one:

```go
app := tork.New(tork.StrictBodies())
```

Then it is reported by name:

```json
{"field": "colour", "issue": "unknown_field", "message": "colour is not a field this endpoint accepts."}
```

## What a bad body reports

Body fields are named by their path inside the document, with no prefix — so a
`billing.zipCode` reads as itself.

| Body | Reported as |
|---|---|
| *(empty)* | `{"field": "body", "issue": "body_required"}` |
| `{oh no` | `{"field": "body", "issue": "invalid_json"}` |
| `[1,2,3]` into a struct | `{"field": "body", "issue": "invalid_type"}` |
| `{"price":"free"}` | `{"field": "price", "issue": "invalid_type", "message": "price must be a whole number."}` |

A type mismatch names the field because `encoding/json` already knows which one
it was and what it wanted, so there is no reason to answer "the body was bad"
and leave the client to work out which part.

The message says what JSON shape was expected — a string, a whole number, a
number, true or false, an array, an object — because that is the vocabulary the
client is working in, not Go's.

## Reported together with everything else

A rejected body and rejected parameters arrive in one answer:

```
PUT /items/42?dryRun=maybe
{"price": "free"}
```

```json
{
  "code": "VALIDATION_ERROR",
  "message": "Validation failed for 2 fields.",
  "status": 400,
  "timestamp": "2026-07-23T16:37:23Z",
  "path": "/items/42",
  "details": [
    {"field": "query.dryRun", "issue": "invalid_boolean", "message": "dryRun must be true or false."},
    {"field": "price",        "issue": "invalid_type",    "message": "price must be a whole number."}
  ]
}
```

A body that cannot be read at all — the wrong media type, or too large — is
different: it ends the request on its own, because nothing else about it would
be meaningful.

## Rules

A body's fields are validated by rules declared with `tork.DefineBody`, which
belong to the body type wherever it is used. See
[Typed declarations](/docs/2.fundamentals/9.typed-declarations).

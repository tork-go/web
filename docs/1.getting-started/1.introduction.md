# Introduction

Tork Web is a Go web framework. Handlers are ordinary functions: they take a
context, one or more structs describing the request, and whatever else they
need; they return a value and an error.

```go
package main

import (
	"context"
	"log"

	"github.com/tork-go/web"
)

type ReadItemInput struct {
	ItemID string `path:"item_id"`
}

type ItemResponse struct {
	ID   string `json:"id"`
	Name string `json:"name"`
}

func readItem(ctx context.Context, in ReadItemInput) (ItemResponse, error) {
	return ItemResponse{ID: in.ItemID, Name: "Boots"}, nil
}

func main() {
	app := tork.New(tork.Title("Example API"))
	app.GET("/items/{item_id}", readItem)

	log.Fatal(app.Run(":8080"))
}
```

Nothing is registered in a container, no interface has to be satisfied, and
there is no `http.ResponseWriter` to remember to write to. The signature itself
is the specification: the framework reads it to learn what to decode, what to
inject, and — once the OpenAPI subsystem lands — what to write into the
document.

## Read once, not per request

It reads that signature exactly once. When the application builds, every
route's handler is classified and compiled into closures: a binder per input
field, an encoder per response type. Tag parsing, name lookup, and type
inspection happen there and never again.

That is what separates this from a reflection-driven framework. The convenience
is paid for at boot, not on every request. The generator planned for the next
milestone replaces those closures with straight-line Go, which removes the last
of the reflection; the runtime stays the definition of what the generated code
has to do.

## Mistakes are build errors

Because everything is checked at build time, a mistake in your source is
reported the way a compiler reports one — with the line you wrote it on, and
with every other mistake found in the same pass:

```
items/router.go:24: GET /items: field ItemID reads the path parameter "item_id",
    but GET /items has no {item_id} in it
items/router.go:31: tork.Summary is not valid on a router
items/handler.go:12: GET /items/{item_id}: parameter 1: field Page has an
    invalid default "lots": it must be a whole number
```

`app.Handler()` returns those as an error, so a test can assert on them without
starting a server.

## What you get

- **Handlers that are just functions.** No framework types in the signature
  unless you want them, and full IntelliSense because everything is typed.
- **Struct-based request binding.** `path`, `query`, `header`, `cookie`,
  `form`, and `body` tags say where each field comes from. See
  [Path parameters](/docs/2.fundamentals/3.path-parameters).
- **Declarations the compiler checks.** `DefineInput` moves names, defaults, and
  validation rules out of tag strings and into typed calls, so a length rule on
  an integer does not compile. See
  [Typed declarations](/docs/2.fundamentals/9.typed-declarations).
- **Routers that own metadata.** A feature package returns one `*Router`
  carrying its prefix and tags, and the application mounts it. See
  [Routers](/docs/2.fundamentals/1.routers).
- **One error shape.** Every failure — yours and the framework's — answers with
  the same envelope, and every bad field in a request is reported in one
  response rather than one at a time. See [Errors](/docs/2.fundamentals/11.errors).
- **Standard library routing.** `net/http.ServeMux` does the matching, so
  `{item_id}` is stdlib syntax and any `func(http.Handler) http.Handler`
  middleware composes unchanged.
- **No dependencies.** The runtime imports nothing outside the standard
  library.

## What is not built yet

Tork Web is being built in steps, and this documentation describes only what
works today. Not yet implemented:

- Dependency injection — `tork.Provide` and `tork.Depends`.
- Typed response wrappers (`tork.Created[T]`, `tork.NoContent`, and the rest).
- OpenAPI documents and the embedded Swagger UI.
- The source generator and its compile-time diagnostics.

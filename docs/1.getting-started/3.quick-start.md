# Quick Start

This builds a small items API: a list endpoint with pagination, a read endpoint,
and an update endpoint that takes a body. It is laid out the way a real project
would be, with the feature in its own package.

## 1. The feature package

Create `internal/items/items.go`. Everything the feature knows about lives here:
its request types, its handlers, and the router that mounts them.

```go
package items

import (
	"context"

	"github.com/tork-go/web"
)

// ItemResponse is what the API returns. The json tags are the wire contract.
type ItemResponse struct {
	ID    string `json:"id"`
	Name  string `json:"name"`
	Price int64  `json:"price"`
}

// ListItemsInput is the query string, as a struct.
type ListItemsInput struct {
	Page   int    `query:"page"  default:"1"`
	Limit  int    `query:"limit" default:"20"`
	Search string `query:"search"`
}

func listItems(ctx context.Context, in ListItemsInput) ([]ItemResponse, error) {
	return []ItemResponse{
		{ID: "1", Name: "Boots", Price: 4200},
	}, nil
}

// ReadItemInput names the wildcard in the route it is used by.
type ReadItemInput struct {
	ItemID string `path:"item_id"`
}

func readItem(ctx context.Context, in ReadItemInput) (ItemResponse, error) {
	if in.ItemID != "1" {
		return ItemResponse{}, tork.NotFound("Item", in.ItemID)
	}
	return ItemResponse{ID: in.ItemID, Name: "Boots", Price: 4200}, nil
}

// UpdateItemBody is the JSON document the client sends.
type UpdateItemBody struct {
	Name  string `json:"name"`
	Price int64  `json:"price"`
}

// UpdateItemInput mixes a path parameter, a query parameter, and the body.
type UpdateItemInput struct {
	ItemID string         `path:"item_id"`
	DryRun bool           `query:"dryRun"`
	Body   UpdateItemBody `body:"json"`
}

func updateItem(ctx context.Context, in UpdateItemInput) (ItemResponse, error) {
	return ItemResponse{ID: in.ItemID, Name: in.Body.Name, Price: in.Body.Price}, nil
}

// Router is what the application mounts. The prefix and tags belong here,
// beside the handlers, not in main.go.
func Router() *tork.Router {
	router := tork.NewRouter(
		tork.Prefix("/items"),
		tork.Tags("items"),
	)

	router.GET("/", listItems, tork.Summary("List items"))
	router.GET("/{item_id}", readItem, tork.Summary("Read an item"))
	router.PUT("/{item_id}", updateItem, tork.Summary("Update an item"))

	return router
}
```

Three things to notice. `ListItemsInput` is a plain struct, and the tags are the
only thing that makes it a request rather than a dependency. `readItem` returns
`tork.NotFound` as an ordinary error. And `Router()` returns a router that has
not been mounted anywhere, so it does not know or care what prefix it will end
up under.

## 2. The application

Create `cmd/api/main.go`:

```go
package main

import (
	"log"

	"github.com/tork-go/web"

	"example.com/project/internal/items"
)

func main() {
	app := tork.New(
		tork.Title("Example API"),
		tork.Description("Example backend built with Tork"),
	)

	v1 := app.Version("v1", tork.Prefix("/api/v1"))
	v1.Include(items.Router())

	if err := app.Run(":8080"); err != nil {
		log.Fatal(err)
	}
}
```

`Include` folds the prefixes together, so `router.GET("/{item_id}", ...)` under
`/items` under `/api/v1` serves `GET /api/v1/items/{item_id}`. A route path of
`"/"` means the prefix itself, which is why `listItems` answers
`GET /api/v1/items` and not `/api/v1/items/`.

## 3. Run it

```sh
go run ./cmd/api
```

The defaults apply when a parameter is left out:

```sh
curl -s 'localhost:8080/api/v1/items'
# [{"id":"1","name":"Boots","price":4200}]

curl -s 'localhost:8080/api/v1/items/1'
# {"id":"1","name":"Boots","price":4200}
```

A body binds beside the path and query parameters:

```sh
curl -s -X PUT 'localhost:8080/api/v1/items/1?dryRun=true' \
  -H 'Content-Type: application/json' \
  -d '{"name":"Wellies","price":3900}'
# {"id":"1","name":"Wellies","price":3900}
```

## 4. What a bad request gets back

The error you returned:

```sh
curl -s 'localhost:8080/api/v1/items/99'
```

```json
{
  "code": "RESOURCE_NOT_FOUND",
  "message": "No item found with ID '99'.",
  "status": 404,
  "timestamp": "2026-07-23T16:37:23Z",
  "path": "/api/v1/items/99",
  "details": {"resource": "Item", "identifier": "99"}
}
```

And the errors the framework found, all of them in one answer — a query
parameter that is not a number and a body field of the wrong type:

```sh
curl -s -X PUT 'localhost:8080/api/v1/items/1?dryRun=maybe' \
  -H 'Content-Type: application/json' \
  -d '{"price":"free"}'
```

```json
{
  "code": "VALIDATION_ERROR",
  "message": "Validation failed for 2 fields.",
  "status": 400,
  "timestamp": "2026-07-23T16:37:23Z",
  "path": "/api/v1/items/1",
  "details": [
    {"field": "query.dryRun", "issue": "invalid_boolean", "message": "dryRun must be true or false."},
    {"field": "price", "issue": "invalid_type", "message": "price must be a whole number."}
  ]
}
```

Every field is attempted before the request is refused, so a client fixing its
call learns everything wrong with it in one round trip. Parameters are named by
where they came from — `query.dryRun` — and body fields by their path inside the
document.

## 5. Test it without a port

`app.Handler()` returns the `http.Handler` without listening, which is what a
test wants:

```go
func TestReadItem(t *testing.T) {
	app := tork.New()
	app.Include(items.Router())

	handler, err := app.Handler()
	if err != nil {
		t.Fatalf("build: %v", err)
	}

	rec := httptest.NewRecorder()
	handler.ServeHTTP(rec, httptest.NewRequest("GET", "/items/1", nil))

	if rec.Code != http.StatusOK {
		t.Fatalf("status %d: %s", rec.Code, rec.Body)
	}
}
```

The same call is where every declaration mistake surfaces. If a route names a
wildcard its input struct does not read, or an option is used somewhere it does
not belong, `Handler` returns it as an error — all of them together, each with
the file and line it was written on.
